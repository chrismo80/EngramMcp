# REQUIREMENTS

## Goal

Build an MCP server for coding agents that provides a simple persistent memory system.

The server is intended to let agents store and recall plain-text memories across sessions without requiring a database, vector store, or any external service.

## Primary Users

- Coding agents using the server through MCP

## Primary Value

- Give coding agents a lightweight memory mechanism they can use across sessions
- Keep the system transparent, local, and easy to inspect by humans
- Prefer deterministic behavior over smart but opaque memory heuristics

## Scope For V1

The first version must provide:

- A single JSON memory file on disk
- Multiple named memory sections inside that file, stored as a name-keyed collection
- Three predefined memory systems:
  - `shortTerm`
  - `mediumTerm`
  - `longTerm`
- A minimal memory entry model with only:
  - `timestamp`
  - `text`
- FIFO eviction when a memory section reaches its capacity
- MCP tools for storing memories into the three predefined sections
- An MCP tool for recalling all stored memories from all sections

## Functional Requirements

### Memory File

- The server must persist all memories in a single JSON file
- The location of this file must be provided when the MCP server is started
- The file path may point to a location outside the current workspace or repository
- If the file does not exist, the server must create it with the expected initial structure
- If the file exists, the server must load its contents on demand

### Memory Structure

- The JSON file must contain multiple named memory sections
- V1 must define exactly these sections in code:
  - `shortTerm`
  - `mediumTerm`
  - `longTerm`
- The persisted structure should use the memory section name as the key rather than fixed dedicated properties in the in-memory model
- The implementation must be generic so that more memory sections can be added later without redesigning the core logic
- Each memory section must have a name and a capacity defined in code
- Capacities are constants in V1, not user-editable through the memory file
- Each memory section should be represented in code as an object that encapsulates its own name, capacity, and section-level behavior

### Memory Entry Model

- Each stored memory entry must contain only:
  - `timestamp`: when the server stored the memory
  - `text`: the plain-text memory content
- The timestamp must be generated by the server, not supplied by the caller
- The text must be stored as plain text without additional semantic structure in V1

### Capacity And Eviction

- Each memory section must enforce its own fixed capacity
- When a new memory is stored and the target section is already full, the oldest existing memory in that section must be removed
- Eviction order must follow FIFO behavior
- Eviction must happen only within the target memory section

### MCP Tools

The server must expose the following tools:

- `store_shortterm`
- `store_mediumterm`
- `store_longterm`
- `recall`

#### Store Tools

- Each `store_*` tool must accept plain-text input to store
- Each `store_*` tool must write to exactly one predefined memory section
- The three store tools must share the same underlying storage and eviction logic
- The only difference between the store tools must be which memory section they target

#### Recall Tool

- `recall` must load the current memory file and return all configured memory sections as a name-keyed result
- The result must preserve section boundaries so callers can distinguish `shortTerm`, `mediumTerm`, and `longTerm`
- `recall` must not merge all memories into a single undifferentiated list

## Configuration Requirements

- The MCP server must accept the memory file location from its startup command
- A typical usage pattern is an MCP client entry such as:

```json
{
  "mcp": {
    "memory": {
      "type": "local",
      "command": [
        "engrammcp --location ~/user/personal-memory.json"
      ]
    }
  }
}
```

- The exact command-line option name may still be refined during architecture, but V1 must support passing a file path at startup
- The server must not require the memory file to live inside the active code workspace

## Architectural Constraints

- The three predefined memory systems must use the same shared models and services
- The implementation must not create separate storage implementations per memory type
- The system should be designed around a catalog of memory objects rather than hard-coded special cases scattered through the codebase
- V1 may keep the memory objects and capacities as constants in code
- The design should allow a future fourth or fifth memory section to be added with minimal structural change

## Non-Goals For V1

The first version does not need to support:

- Databases of any kind
- Vector storage or embeddings
- Semantic search
- Automatic promotion between memory sections
- Memory updates or targeted deletion of specific entries
- User-managed capacities inside the memory JSON file
- Dynamic user-defined memory section definitions from external configuration
- Rich metadata such as tags, IDs, relations, or scopes

## Error Handling Requirements

- The server must fail clearly if the configured memory file path is invalid or inaccessible
- The server must fail clearly if the memory file contains invalid JSON
- The server must avoid silently corrupting or overwriting memory data structure on malformed input
- Error responses should make it clear whether the failure comes from configuration, file access, or JSON format issues

## Quality Requirements

- Behavior must be deterministic and easy to reason about
- The stored JSON should remain human-readable
- The system should be straightforward to inspect and debug manually
- The implementation should stay intentionally simple and avoid speculative abstractions beyond the generic memory-definition model

## Acceptance Criteria

V1 is considered complete when:

- Starting the server with a memory file path works for a file inside or outside the workspace
- The server creates the memory file if it does not already exist
- `store_shortterm` appends a new memory into `shortTerm` and enforces FIFO eviction
- `store_mediumterm` appends a new memory into `mediumTerm` and enforces FIFO eviction
- `store_longterm` appends a new memory into `longTerm` and enforces FIFO eviction
- `recall` returns all configured memory sections separately
- All three memory sections use the same underlying storage and memory-management logic
- The persisted file contains only the configured memory sections and entries with `timestamp` and `text`, keyed by memory section name
